# AI 时代，什么能力才真正重要

## 分工变了

在 AI 时代，传统"手写代码 → 项目落地"的模式正在被重新定义：

| 传统分工 | AI 时代分工 |
|---------|------------|
| 手写代码 → 项目落地 | **提需求 + 把关 → AI 自动写** |
| 精通一门语言 → 有饭吃 | **懂基础逻辑 → AI 帮你补细节** |
| 背 API 记语法 → 熟练工 | **会描述问题 → AI 给你方案** |

**真正重要的不再是打字速度或语法记忆了。**

## 真正重要的四件事

**1. 判断 AI 写的对不对**

能分辨 AI 的输出是靠谱还是胡说。AI 会自信地给出错误答案，没有基本的逻辑判断力就会被带偏。

**2. 把模糊的需求拆成清晰的指令**

AI 是优秀的执行者，但不是优秀的产品经理——它不会替你想清楚"你要什么"。能把需求说清楚的人，在 AI 时代越来越值钱。

**3. 知道什么能做、什么不能做**

理解技术和业务的边界。有些事 AI 做不了、不该做，有些事技术上不可行——这些判断只能靠人。

**4. 自己动手调试和验证**

AI 写代码很快，但 debug 还得靠人。AI 在 debug 场景下的表现远不如写代码，逻辑推理和排查问题仍然是人的核心优势。你在 BASIC 死循环里磨出来的那种"一步步跟踪、找出哪里不对"的能力，恰恰是 AI 时代最值钱的底层技能。

> ⚠️ 上面第 1 点和第 4 点紧密相关。"判断 AI 对不对"不能只靠直觉，要靠亲手验证——跑一下、测一下、断点跟踪一下。**判断力来自动手能力，而不是来自感觉。**

## "编程能力"和"网络知识"本是一体

不存在二选一。现代编程几乎都离不开网络——脚本要调 API、网站要处理请求、工具要连数据库。**学编程的过程中顺便理解网络**就行，不需要深钻，但最低限度的理解很有帮助：

- **HTTP 请求**是怎么回事——你打开浏览器、AI 去调 API，背后都是这个
- **客户端/服务器模型**是什么
- **API** 是什么——你用 CC Switch 就是在用 API 代理

懂了这些之后，AI 给你的代码里很多"莫名其妙"的东西就变得合理了。

## 建议的学习路径

基于你的背景（85 年 BASIC、现用 AI 工具），最实用的路线：

**1. Python 基础**

- 变量、条件、循环——你在 BASIC 里都接触过，只是换了个写法
- 重点理解**"逻辑"而不是语法**，AI 会帮你写语法

**2. 学会用 AI 写代码**

- 你已经在用 Claude Code 了，这本身就是答案
- 多练**"把想法翻译成提示词"**，这是新时代的核心技能

**3. 理解计算机联网的基本原理**

- 知道 F12 开发者工具能看到什么
- 理解你正在用的 API 是怎么工作的

## "65 分"的真相——够用，但有盲区

技术行业有个说法：**80% 的人只需要 65 分的编程能力**。能把 Excel 宏写明白、能写个脚本自动处理文件、能搭个简单网站——这就已经超过大多数人了。你从 BASIC 摸爬滚打出来的逻辑思维，已经具备了这 65 分的基础。

但 65 分有它的盲区。最大的危险不是你不会，而是 **你意识不到自己遇到了那 20% 的棘手场景**：

- AI 写的代码有**性能隐患**（循环套循环处理大量数据），你只觉得"能跑就行"
- AI 生成的代码有**安全漏洞**（直接把用户输入拼进 SQL），你不知道这是漏洞
- AI 给出的方案在**数据量大了以后会崩**，你看不出架构上的瓶颈

**应对方法不是非要考到 100 分**，而是建立两条意识：

1. **知道自己能力的边界**——遇到涉及安全、性能、大规模数据、钱的场景，多留个心眼
2. **知道什么时候该找专业的人**——能判断"这个问题可能比我想象的复杂"，然后去问、去查、去找人 review

> 65 分的技能 + 对边界的清醒认知 = 远比 65 分强大。

## 你的行业经验才是"护城河"

"懂基础逻辑 → AI 帮你补细节"这个流程有个关键前提：**你懂这个领域的常识**。

如果让 AI 写一个支付功能，它不会自动考虑对账、幂等性、事务边界——不是它不能，而是你没告诉它这些是需求的一部分。**你的"基础逻辑"里需要包含对问题域的基本认知，而不仅仅是编程语法。**

你在现实世界中积累的行业经验（业务流程、管理逻辑、实际场景里踩过的坑），恰恰是 AI 补不了的部分。这些才是你作为人类的"护城河"。
