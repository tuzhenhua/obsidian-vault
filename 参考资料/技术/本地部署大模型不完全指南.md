# 本地部署大模型不完全指南

## 第一部分：背景——大模型/Agent 的成熟

自从以 Claude Code 为代表的编程 Agent 成熟起来以后，coding 场景就成为大模型的重度使用场景。而由于特殊的情况，又衍生了被称为御三家的**细粮**，和这边的阿里千问、智谱 GLM、月之暗面 Minimax（DeepSeek V4 刚发布，小米的感觉在 coding 场景的重度用户不多）等**粗粮**。细粮的意思是细节更完善，使用更丝滑，意图理解更深入完整，输出质量更高。这其中模型的水平占据一部分影响，接下来就是 Agent 和其插件等构成的工程能力，以及和模型的配合密切程度了。

从软件工程角度讲，如今的编程 Agent + 大模型，已经将软件开发带入了**面向 AI 的软件工程**的时代，也就是已经超越了 vibecoding 层次了。而相对应的，模型提供商也迎来了他们的春天时代。随着 token 使用量的迅速上升，有一个很重要的变化就是这边的 coding plan 逐步被取消，转而以 token 计费方式为主。这固然其中有部分因素是由于以 OpenClaw 为代表的龙虾类 Agent 对于 token 使用的毫无节制，也在于在 coding Agent 场景下，基本也是使用的大模型的深度思考模式，以及在 Agent 中也是以 plan 方式为主——先完成详细的代码草稿，审阅并定稿，再交给 Agent 执行。这其中 token 的消耗也是远远高于日常的简单提问对话的。

那边的御三家固然好，但是要么购买转发调用的，要么拼套餐，当然了，如果你们公司给直接开通无限量，那最好了。但是费用上涨，尤其是 coding plan 类的收紧或者取消，毫无疑问会给个人使用者带来不小的影响。这种情况之下，**能否使用本地部署模型来进行 coding？**

另一个促使本地部署模型的动因就是**隐私问题**。在使用厂商/API 服务商的服务时，本地的信息会大量地被喂给模型——不管是代码还是个人的其他工作产物。目前这个方面是属于灰色地带的。

还有一个场景就是对于使用 OpenClaw 类的个人工作处理类 Agent 时，其带来的高消耗量，以及可能涉及到的敏感信息安全，也会考虑需要本地部署。

在这个部分的最后，需要说明下：**本文不会涉及任何使用 Ollama 的内容**。本文是面向工程化视角，面向在实际的任务背景下本地场景部署大模型的，并在网络访问等条件约束下，直接使用推理框架部署，更为直接。

## 第二部分：本地部署的模型选择

本文中的本地部署，特指个人在日常办公电脑尤其是笔记本电脑的部署，其中的内容也适用于台式机。所以针对于购买专门的 AI 算力服务器，不在本文讨论之列，参考主流推理引擎的部署指导即可。

个人部署时面临的第一个问题就是**硬件资源受限**，所以模型的选择就比较关键。好在世界总有带回给你光芒的人。在大模型的能力越来越成熟、参数量越来越高的背景下，仍然有厂商没有忘记个人用户，发布了同代小参数量的模型。从目前能够获取的开源模型来看，**Qwen3.5、Qwen3.6 和 Gemma 4** 是可供选择的。

### 模型基本信息对比

| 维度 | Qwen3.5 | Qwen3.6 | Gemma 4 |
|------|---------|---------|---------|
| **发布方** | 阿里巴巴·通义实验室 | 阿里巴巴·通义实验室 | Google DeepMind |
| **发布时间** | 2025 Q4 | 2026 Q1 | 2026 H1（预测） |
| **模型类型** | 稠密 + MoE 混合 | 稠密 + MoE 增强架构 | MoE 轻量架构 |
| **参数规格** | 0.5B / 3B / 7B / 9B / 27B / 35B-A3B / 122B-A10B / 397B-A17B | 27B / **35B-A3B** / 满血版未开源 | E2B（5.1B-A2）/ E4B（8B-A4.5）/ 26B-A4B / 31B |
| **专家总数** | 256 | 256-512 | 256 |
| **激活专家数** | 10 个路由专家 + 1 个共享专家 | 10 个路由专家 + 1 个共享专家 | 8 |
| **上下文长度** | 27B 及以上 **256K** | 256K | 256K |
| **支持语言** | 100+（中英优化） | 100+（中英增强） | 40+（英文优先） |
| **开源协议** | Apache 2.0 | Apache 2.0 | Gemma Terms v2（商用需注册） |

### 模型大小与量化后体积

个人本地部署时，因为内存的限制，需要选择小参数量的版本。以 Qwen3.5 为例，不同版本的大小信息如下：

| 版本 | 架构 | 总参数 | 激活参数 | 上下文 | 原始大小（FP16） | Q4_K_M 大小 |
|------|------|--------|---------|--------|:---:|:---:|
| **27B** | MoE | 27B | - | 256K | **~54 GB** | **~19 GB** |
| **35B-A3B** | MoE | 35B | ~3B | 256K | **~70 GB** | **~23 GB** |
| **122B** | MoE | 122B | ~10B | 256K | **~245 GB** | **~88 GB** |
| **397B** | MoE | 397B | ~17B | 256K | **~810 GB** | **~285 GB** |

可以看出，原始的 FP16 大小，对于 122B 和 397B 来说，普通单机的显卡显存是很难容纳的。即使是 Q4_K_M 量化后，也只有 35B 和 27B 存在单张显卡容纳权重的可能性。

### KV-Cache 显存计算

推理时上下文长度和 KV-Cache 的大小正相关。计算公式为：

```
mem_KV = 2 × batch_size × seq_len × (hidden_size / num_heads) × num_kv_heads × num_layers × dtype_bytes（byte）
```

其中 `batch_size` 是推理时批次大小，是可调的，最小为 1（仅服务单个用户）；`seq_len` 是输入和输出的总长度；`dtype_bytes` 是进行量化处理时的大小。

以 Qwen 27B 为例，设本地部署场景下用到最大上下文 256K 时（极限），KV-Cache 大小约 **48GB**（BF16）。以 32K 上下文计算时也有 **6GB**。如果进行 Q8 量化，则需要 3GB。

### 模型选择结论

就模型的性能和大小的考虑而言，对于个人部署的场景，Q4 量化后在 **24GB 以下为好**，同时模型的参数量不能太小。所以可选择的范围就是 Qwen 系列的 **35B、27B** 和 Gemma 4 的 **26B、31B**。

接下来需要说明的是：Qwen 的 35B 是 MoE 架构，27B 是 Dense 架构；而 Gemma 4 恰好相反，26B 是 MoE 的，31B 是 Dense 的。MoE 和 Dense 之间的选择，主要考虑显存是否充足。因为 MoE 架构的模型是部分激活的（模型的后缀 `AxB` 就是激活 XB 的专家权重），在显存不能完全容纳、需要卸载到 CPU/内存时，受到的影响会更小。

由于 Qwen3.6 版的 27B/35B 已经开源，所以在 Qwen 系列选择中应当选择 **3.6 版本**。而在 Gemma 4 和 Qwen3.6 的选择中，由于 Qwen 的旗舰版本是 397B 的，那么其 35B/27B 版本在训练数据集和技术上会具有同源性，性能上限会高于纯小模型 Gemma 4，而基准测试也表明了这一点。

## 第三部分：硬件方案

对于大模型的推理而言，毫无疑问，**算力和显存大小**是最为关键的。而在个人的笔记本上部署，恰恰这两个也是最受限的，并且难以同时得到最优解。

### GPU 选择

先说显卡——**NVIDIA / Intel / AMD**：

- **Intel / AMD**：新款集成显卡才具备基本的本地部署算力——Intel 的酷睿 Ultra 系列 388H/358H/338H 处理器，AMD 的 AI Max 395/392/390/385 处理器。其中单独计算 GPU 算力的话，Intel 处理器的 GPU 算力领先于 AMD 的同档次处理器。
- **NVIDIA**：肯定是独立显卡了，基本上 RTX 30 系列的算力就能满足需要。

### 显存问题

对于笔记本电脑来说，如果是 NVIDIA 显卡，显存以 **8GB 或更少**居多，高于 8GB 显存的可以说是偏高档的机器。但是大模型对于显存的需求可以说远大于游戏。所以对于独显的笔记本电脑来说，显存问题的解决：
- 购买大显存（16GB）的型号
- 选择 AMD AI Max 或 Intel 3x8H 处理器的笔记本
- 针对 RTX 30 系列显卡，到专业门店更换显存颗粒，将显存从 8GB 升级到 16GB

### 统一内存技术

AMD AI Max 或 Intel 3x8H 处理器可以说是针对于 AI 潮流而推出的。诚然，它们的图形处理能力相比以前有了大幅度的提升，但是和 NVIDIA 相比，如果为了游戏或者做图而选择这两种类型的笔记本，只能说仅够可用。

它们都具备**统一内存技术**——显示核心和 CPU 核心共用内存控制器 MMU 单元，高速访问系统内存，把系统内存作为显存。例如：

- 系统内存 128GB → AMD 最多分配 **96GB** 作为显存（3/4）
- Intel 3x8H 平台最多可分配 **87%** 的内存作为显存（操作系统需要保留足够内存给自己和应用使用）

#### 价格现状

统一内存机型定位以及内存价格的走高，让相关产品价格居高不下：

| 配置 | 参考价格 |
|------|---------|
| AI Max 32GB | ~9,000 元 |
| AI Max 64GB（华硕系列） | ~13,000 元 |
| AI Max 128GB（幻系列） | ~23,000 元 |

如果选择 32GB 的内存配置，则需要进行低比特量化，否则系统内存会很紧张。

Intel 3x8H 平台方面情况也很不乐观：虽然其 GPU 算力很好，但是芯片产能感人，供应量少，推出的产品内存最大也只有 32GB 版本。其中的高配机型（如 ThinkBook Pro 的 358H 版本）价格和 AMD AI Max 相差无几，且没有更大内存的机型可选。

### 硬件与模型部署方案总表

| 显卡类型 | 显存 | 可用部署方案 | 备注 |
|---------|:---:|------------|------|
| NVIDIA 独立显卡 | 8GB | 板载显存扩容到 16GB | 需要到专门店铺扩容显存 |
| NVIDIA 独立显卡 | 8GB | 35B 量化（≤Q4）混合推理 | 将 MoE 专家卸载到内存，计算通过 CPU 进行 |
| NVIDIA 独立显卡 | 16GB | 35B 量化（≤Q4）混合推理 / 27B 量化（<Q4）完全显存推理 | |
| AMD AI Max / Intel 3x8H | 32GB | 35B 量化（Q4）完全显存推理 / 27B 量化（Q4）完全显存推理 | 统一内存 |
| AMD AI Max | 64GB | 35B 量化（≥Q4）完全显存推理 / 27B 量化（≥Q4）完全显存推理 | 统一内存 |
| AMD AI Max | 128GB | 35B 完全显存推理 / 27B 完全显存推理 / 122B Q4 量化推理 | 统一内存 |

## 第四部分：量化方案选择

在第三部分中，除了 128GB 顶配场景，其它场景均涉及量化。当然，128GB 场景仍然可以对大尺寸模型进行量化。在目前，由于量化水平的进步，Q4 量化对于精度损失已经很小了。新的 **Q4_K_M** 量化采用的是动态量化，对激活部分采用更高比特量化，并使用校准过程识别权重的重要程度，已经十分成熟。

对于显存更为受限的配置，量化可以说是必须进行的。根据目前的信息综合而言，优选的量化类型包括 **Q_K_M、IQ/IQ_KT 和 EXL3** 三种。至于通常的 INT8，体积太大，量化收益太低；INT4 性能太差，本文不做讨论。

### 量化技术对比

| 量化类型 | 举例 | 核心处理逻辑 | PPL / 特点 |
|---------|------|-------------|-----------|
| **Q_K_M** | Q4_K_M | 细粒度分块 + 非线性码本 + 多级缩放。技术升级：① 分块策略：256 weights/block（比 Q4_0 的 32 更细）；② 缩放因子：每块 4 个缩放因子（而非 2 个），适应局部分布；③ 量化码本：使用 K-means 聚类优化的非线性码本；④ 精度保留：对极端值使用更高精度编码 | 比通常 Qx_0 具有好得多的 PPL；一般作为同等 Bit 量化的基准，支持成熟 |
| **IQ** | IQ4_XS / IQ4_XSS / UD_IQ_XSS | 在 Q_K_M 基础上通过数据集产生校准数据，进行权重的重要性感知校正 | 同体积下 PPL 更低（好于 Q_K_M），速度会降低少许 |
| **IQ_KT** | IQ4_KT | 在 IQ 基础上针对现代加速器硬件优化内存布局与解码流程：① 权重按张量核心的 16×16/32×32 块对齐；② 缩放因子与权重数据连续存储，减少内存事务；③ 减少条件分支，提高分支预测命中率；④ 批量查表提升吞吐量；⑤ 关键层自动使用更高精度编码，非关键层使用激进压缩 | PPL 和 IQ 相当，体积略大，速度提高 |
| **EXL3** | 无约定命名规则 | 基于网格编码量化（TCQ）的向量量化方法，通过不相关预处理使权重分布近似高斯，再用维特比最优编码在超高维空间实现接近理论极限的压缩效率 | 同体积具有最好的 PPL 表现，远超 Q_K_M |

基于上面的表格，再结合部署机器的显存大小，就可以在部署时进行模型/量化的选择。

## 第五部分：推理框架选择

个人部署场景下，即使具备了 128GB 的显存/内存，和大规模高并发推理系统相比，仍然属于资源受限环境。在推理框架的选择上，这里就不去介绍 vLLM 和 sglang 的相关信息了，着重针对不同模型格式介绍相关的框架。

Q_K_M / IQ / IQ_KT 的推理框架都属于 **llama.cpp 体系**，EXL3 则使用专用的 **ExLlamaV3** 推理引擎。

### 推理框架对比

| 框架名称 | 概况 | 模型格式 | 特点 | 仓库 |
|---------|------|---------|------|------|
| **llama.cpp** | 主线版本 | GGUF | 更新快，混合推理，成熟，使用最广 | [github.com/ggml-org/llama.cpp](https://github.com/ggml-org/llama.cpp) |
| **ik_llama.cpp** | 性能优化版，从主线 fork | GGUF | 跟随主线，针对 CUDA 做算子的特别优化实现，性能更高 | [github.com/ikawrakow/ik_llama.cpp](https://github.com/ikawrakow/ik_llama.cpp) |
| **ik_llama.cpp（Thireus 版）** | IQ_KT 特别版，从 ik_llama.cpp fork | GGUF | 在 ik_llama.cpp 基础上完善对 IQ_KT 支持 | [github.com/Thireus/ik_llama.cpp](https://github.com/Thireus/ik_llama.cpp) |
| **ExLlamaV3** | EXL3 专用 | EXL3 | 使用专门的 EXL3 格式，不能进行混合推理 | [github.com/turboderp-org/exllamav3](https://github.com/turboderp-org/exllamav3) |

### 选择建议

- 如果硬件资源充足、机器配置好、显存充分的情况下，可以选择 **llama.cpp**。
- 如果一个模型同时具备 Dense 和 MoE 模型（如 Qwen3.5-27B / Qwen3.5-35B），在显存充分的情况下**优先选择 Dense 模型**——从基准评测看，Dense 模型的分值更高。
- 如果显存不充分，对于 NVIDIA 显卡，选择 **MoE 模型**，将专家卸载到 CPU/系统内存中。
- 对于 AMD/Intel 的统一内存场景，除了 128GB 显存之外的时候，可以自由选择 Dense 或者 MoE 模型。
- 对于资源受限又希望体验高性能的小伙伴，可以尝试下 **ExLlamaV3**。它目前已经支持 Qwen3.5，但因为过于小众，模型的量化需要自行进行，且不支持混合推理，也不支持除了 CUDA 之外的其他生态。

### 部署参数注意事项

注意在进行模型部署时，需要仔细进行参数设定，包括对话模板、RoPE 等参数，以提升表现。

使用 llama.cpp 及变体部署时，如果使用 MoE 模型并且使用 NVIDIA 显卡，注意参数设置时先将所有 MoE 专家卸载到 CPU/内存，使用 `-ngl 999` 加载所有层到 GPU，然后根据剩余的显卡显存空间，逐步减小卸载的 MoE 参数值，以充分利用显存，提升推理速度。Intel/AMD 平台因为是统一内存，不涉及此问题。

## 第六部分：附加信息——NVIDIA 显存的软件扩充 / KV-Cache 量化及可能的进化方向

对于个人本地部署而言，特别是移动办公场景，在笔记本电脑上部署大模型，在有了量化、llama.cpp 的混合推理加持之下，**显存的矛盾仍然是主要矛盾**。因为第一，不可能每个人都去购买一台 128GB 的 AMD AI Max 笔记本电脑；Intel 的 3x8H 系列高配机型很少，产能不够；AMD 的 ROCm 生态不如 CUDA 成熟，Intel 的生态同样。所以除了价格问题之外，CUDA 生态的成熟性也让基于 NVIDIA 显卡进行部署成为最常用的方案——然而 **NVIDIA 不支持统一内存**。

### GreenBoost：把系统内存当显存用

为了解决 NVIDIA 不支持统一内存，无法直接借助系统内存扩充显存的问题，一位开发者开发了 [GreenBoost](https://gitlab.com/IsolatedOctopi/greenboost)，来让 NVIDIA 显卡可以把系统内存当作显存使用（不是 NVIDIA 的 Unified Memory）。其基本原理是使用 DMA-BUF 或 `hostMemReg` 将系统内存注册给显卡，显卡直接通过 PCIe 总线读写系统内存。原来这种特性是用于 GPU 和其他子系统高效交换数据的（绕过显存-内存拷贝），但是结合操作系统的巨页等特性，的确能够让显卡"看到"更多的内存。

前面提到，llama.cpp 在进行混合推理时，卸载到 CPU/内存中的专家在参与推理时，算子运算是在 CPU 上进行的。而 CPU 进行矩阵类运算，即使有优化，也不能和 GPU 相比。

这个技术目前 llama.cpp 还没有进行适配，因为系统内存注册成的显存，带宽受限于 PCIe 带宽，在 PCIe 3.0/4.0 还是要比显存低很多。所以需要 llama.cpp 能够主动将 MoE 专家、KV-Cache 放置在这种"内存显存"中，以降低对速度的影响。而 **ExLlamaV3 有对应的 KV-Cache 压缩组件，能够充分利用该特性**。详情请参考 GreenBoost 文档。但是使用 ExLlamaV3 需要自己进行量化处理。

> 个人曾经修改过 llama.cpp 的代码，使用 Dense 模型，将权重拆分成两部分，比卸载一部分到 CPU/内存要快，但是对于 MoE 型，需要单独适配，没有进一步尝试。

### KV-Cache 量化

对于实际的生产性部署，长上下文是不得不面对的。此时需要考虑对 KV-Cache 使用量化。llama.cpp 自带支持对 KV-Cache 做 Q4、Q8 量化。

而目前最新的 **TurboQuant** 量化，从原理上可以和低比特量化结合。目前有开发者在 fork 分支上增加了 TurboQuant 的支持，合入 llama.cpp 主线的事宜在热烈讨论中。

所以，如果 llama.cpp 能够对 GreenBoost 适配，或者自身直接支持将内存注册为显存使用，以及增加对于 TurboQuant 的支持，无疑会非常好。

## 附录 A：本地部署的经济账——值不值？

### 一个常见的误解

**"本地部署能省 token"**——这是不对的。本地部署不会减少 token 的消耗量，该处理的上下文、该生成的代码，计算量完全一样。本质区别在于**成本模型不同**：

| | 云端 API | 本地部署 |
|---|---|---|
| **成本模型** | 按量计费，越用越贵 | 一次性硬件投入，边际成本趋近于零 |
| **适用场景** | 低频使用、追求效果 | 高频重度使用、隐私敏感 |
| **隐私** | 代码和数据全上传 | 数据不出本机 |

打个比方：**打车 vs 买车**。天天打专车，最后算下来可能还不如买一辆——前提是你跑得足够多。

### 当前云端 API 的真实成本（以 DeepSeek 为例）

截至 2026 年 5 月，DeepSeek 经历大幅永久降价后，定价如下（单位：元/百万 token）：

| 模型 | 输入（缓存命中） | 输入（缓存未命中） | 输出 |
|------|:--:|:--:|:--:|
| **V4-Flash**（轻量） | 0.02 | 1 | 2 |
| **V4-Pro**（旗舰） | 0.025 | 3 | 6 |

#### 一次典型重度 coding agent 任务的费用估算

假设输入 50 万 token（代码上下文 + 对话历史），输出 20 万 token（推理 + 生成代码）：

| 模型 | 单次任务费用 | **100 元能用多少次** |
|------|:--:|:--:|
| **V4-Flash** | ≈ 0.60 元 | **≈ 160 次**（每天 5 次以上） |
| **V4-Pro** | ≈ 2.70 元 | **≈ 37 次**（每天 1 次以上） |

100 元/月在 DeepSeek 这边，对绝大多数个人开发者完全是够用的。

### 本地部署的真实成本 vs 云端 API

| | 本地部署路线 | 轻薄本 + API 路线 |
|---|---|---|
| **硬件投入** | 1~2.3 万元（AI Max / 高配独显） | 随便一台轻薄本即可 |
| **模型效果** | Qwen 35B Q4，有精度损失 | DeepSeek V4 / Claude 满血模型 |
| **长期费用** | 电费 | ~100 元/月 |
| **同等硬件预算可用 API 多久** | — | **约 8~19 年** |
| **隐私** | 数据不出本机 | 代码上传云端 |
| **折腾程度** | 量化调参、框架适配、GreenBoost… | 装个终端工具就开用 |

### "一端重一端轻"架构

coding agent 本质上是一个调度器——读文件、发 prompt、收结果、写回文件，这些对本地硬件几乎没要求。真正吃算力的模型推理全在云端：

```
轻薄本（轻）                  云端（重）
├─ Agent 逻辑运行            ├─ 大模型推理
├─ 代码编辑、文件操作         ├─ 深度思考、代码生成
└─ 发送/接收 API 请求        └─ 全部 GPU 算力
```

### 结论：谁适合本地部署？

本地部署的**核心驱动力不是省钱，而是两个极端场景**：

1. **隐私敏感到骨子里的**——代码绝不能离开本机，再便宜也不行
2. **用量大到云费用扛不住的**——每天几十上百次 agent 任务的公司/团队级别用量

对绝大多数个人开发者而言，**一台轻薄本 + DeepSeek / Claude API，就是目前性价比最高的方案**。与其掏一两万买硬件，不如先把这钱充 API——足够用很多年了。

## 附录 B：DeepSeek V4-Pro 与 V4-Flash 差距对比

*云端 API 路线下，选哪个模型更合适？*

### 基本参数

| | V4-Pro | V4-Flash |
|---|---|---|
| **总参数** | 1.6T（万亿） | 284B |
| **激活参数** | 49B | 13B |
| **上下文** | 1M | 1M |
| **推理速度** | ~29 t/s | **~103 t/s**（3.5× 快） |
| **输出价格** | 6 元/百万 token | **2 元/百万 token** |

### 编程基准差距

| 测试 | Pro | Flash | 差距 |
|------|:--:|:--:|:--:|
| **SWE-bench Verified**（修 Bug） | 80.6 | 79.0 | **仅 1.6 分** |
| **LiveCodeBench**（竞赛编程） | 93.5 | 91.6 | **仅 1.9 分** |
| **Terminal Bench 2.0**（多步 Agent） | 67.9 | 56.9 | **11 分 ⚠️** |
| **Apex**（高级推理） | 38.3 | 33.0 | 5.3 分 |

### 解读

- **日常编码（函数生成、写测试、代码补全）** → Flash 完全够用，差距 1-2 分基本不可感知，速度反而更快
- **复杂多步 Agent（跨文件重构、10+ 次工具调用）** → Pro 优势明显，Terminal Bench 11 分差距意味着 Flash 在长链路任务中容易"走丢"
- **最佳策略**：90% 走 Flash + 10% 走 Pro 分层路由，性价比最优

## 附录 C：DeepSeek 替代 Claude（Opus / Sonnet / Haiku）的模型配置

*当使用 Claude Code / OpenClaw 等编程 Agent 接入 DeepSeek 时，需要正确映射三个模型角色。*

### 模型映射

| Claude Code 角色 | 映射到 DeepSeek 模型 | 用途 |
|---|---|---|
| **Opus**（最强推理） | `deepseek-v4-pro[1m]` | 架构设计、复杂重构、安全审计 |
| **Sonnet**（主力干活） | `deepseek-v4-pro` | 日常编码、代码生成执行 |
| **Haiku**（轻快便宜） | `deepseek-v4-flash` | 子 Agent、文件扫描、简单任务 |

> Opus 和 Sonnet 底层都是 V4-Pro，区别仅在于 Opus 启用 1M 超长上下文（`[1m]` 后缀）。

### 环境变量配置

**Windows PowerShell（推荐）：**

```powershell
$env:ANTHROPIC_BASE_URL = "https://api.deepseek.com/anthropic"
$env:ANTHROPIC_AUTH_TOKEN = "sk-你的DeepSeek API Key"

$env:ANTHROPIC_DEFAULT_OPUS_MODEL = "deepseek-v4-pro[1m]"
$env:ANTHROPIC_DEFAULT_SONNET_MODEL = "deepseek-v4-pro"
$env:ANTHROPIC_DEFAULT_HAIKU_MODEL = "deepseek-v4-flash"
$env:ANTHROPIC_MODEL = "deepseek-v4-pro[1m]"
$env:CLAUDE_CODE_EFFORT_LEVEL = "max"
```

**macOS / Linux（~/.zshrc）：**

```bash
export ANTHROPIC_BASE_URL="https://api.deepseek.com/anthropic"
export ANTHROPIC_AUTH_TOKEN="sk-你的DeepSeek API Key"
export ANTHROPIC_DEFAULT_OPUS_MODEL="deepseek-v4-pro[1m]"
export ANTHROPIC_DEFAULT_SONNET_MODEL="deepseek-v4-pro"
export ANTHROPIC_DEFAULT_HAIKU_MODEL="deepseek-v4-flash"
export ANTHROPIC_MODEL="deepseek-v4-pro[1m]"
export CLAUDE_CODE_EFFORT_LEVEL="max"
```

### 实际运作流程

```
你发起一个 agent 任务
    │
    ├─ 主 Agent 规划/执行  →  deepseek-v4-pro      (Opus/Sonnet 级)
    │
    ├─ 子 Agent 读文件     →  deepseek-v4-flash     (Haiku 级)
    ├─ 子 Agent 代码探索   →  deepseek-v4-flash
    └─ 子 Agent 跑测试     →  deepseek-v4-flash
```

### 避坑指南

| 常见问题 | 说明 | 解决 |
|---------|------|------|
| **不支持图片** | V4 是纯文本模型，截图看不到 | 有图片任务临时切回 `claude-sonnet-4-6` |
| **必须加 `[1m]`** | 不加只有 200K 上下文 | Opus 一定用 `deepseek-v4-pro[1m]` |
| **认证字段名** | 是 `ANTHROPIC_AUTH_TOKEN`，不是 `ANTHROPIC_API_KEY` | 严格按上面写 |
| **模型名写错不报错** | 会静默降级到 Flash | 配完去控制台确认实际调用模型 |
| **Base URL 不加 `/v1`** | 加了返回 404 | 保持 `https://api.deepseek.com/anthropic` |
| **旧端点即将退役** | `deepseek-chat` 和 `deepseek-reasoner` 于 **2026年7月24日** 下线 | 迁移到 `deepseek-v4-pro` / `deepseek-v4-flash` |
